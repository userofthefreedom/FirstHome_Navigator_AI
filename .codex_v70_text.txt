0. v7.0 작성 의도와 읽는 법
v7.0은 제품 방향을 새로 바꾸는 문서가 아니다. v6.6에서 확정한 “공식 공고문 기반 주택형 옵션/자금 준비 코치” 정체성을 유지하면서, 프로젝트 구조 정리, 백엔드 라우팅 분리, Vue + JavaScript + Vite 기준 프론트엔드 정리, 마감 공고 제외, 100점 추천 점수 체계, 상세/자금 화면 안정화 내용을 팀 공유 기준으로 재정리한 버전이다.
1~5장은 제품 판단 영역이다. 팀이 무엇을 만들고 무엇을 만들지 않을지 맞춘다.
6~14장은 구현 공유 영역이다. 화면, API, 데이터, LH 수집, PDF 분석, 추천/자금, AI 구조를 이해한다.
15~17장은 실행 영역이다. 현재 완료 상태와 다음 스프린트 과제를 결정한다.
18~20장은 발표, Definition of Done, 명령/파일 체크리스트다.
v7.0의 한 문장
FirstHome Navigator AI는 청약 신청을 대신하는 서비스가 아니라, 실제 LH 공고와 공식 공고문 PDF를 주택형 옵션·납부 일정·내 자금 부족액·다음 행동으로 번역해주는 첫 집 준비 코치다. v7.0에서는 코드 구조와 화면/API 흐름을 팀이 이해하기 쉬운 형태로 정리해, 기능 확장보다 유지보수성과 검증 가능성을 높였다.
1. Executive Summary: v7.0 기준 제품 정의
항목
v7.0 정의
서비스명
FirstHome Navigator AI
한 줄 정의
사회초년생이 공식 공고문을 직접 해석하지 않아도, 소유형 공공분양 공고 안의 주택형 옵션·납부 일정·자격 확인 항목을 내 자금 상황과 연결해 이해하도록 돕는 첫 집 준비 웹서비스.
핵심 고객
공공분양을 처음 검토하는 사회초년생, 청년, 신혼 전 단계 사용자. 청약홈/LH 공고문을 읽어야 한다는 것은 알지만 주택형별 금액, 계약금, 중도금, 잔금, 할부금, 자격 조건을 한 번에 이해하기 어려운 사람.
핵심 문제
LH/API 목록은 공고 단위 정보와 일부 공급정보를 제공하지만, 실제 의사결정에 필요한 납부 구조는 PDF 표와 주석에 흩어져 있다. 또한 마감 지난 공고가 추천에 섞이면 사용자는 지금 검토 가능한 후보를 판단하기 어렵다.
핵심 해결
실제 LH 데이터를 우선 수집하고, 현재 검토 가능한 소유형 공공분양만 추천/대시보드/캘린더에 반영한다. 공식 PDF 분석과 공급정보 API를 통해 주택형 옵션과 납부 일정을 구조화하고, 사용자 자금 상황과 연결해 옵션별 자금 로드맵을 보여준다.
AI의 역할
AI는 확정 판정자가 아니라 공식 근거 기반 설명자다. 부족액, 납부 일정, 확인 항목, 이번 주 행동을 짧고 안전하게 설명하며, 추출 실패/애매한 공고에는 LLM 보조 분석을 적용할 수 있다.
v7.0의 의미
기능 추가보다 프로젝트 형식과 구조를 정리한 버전이다. 백엔드 라우팅은 app별 urls.py로 분리했고, 프론트엔드는 Vue + JavaScript + Vite 기준으로 정리했으며, 추천 점수는 정책 점수 없이 자격 35 + 자금 25 + 지역 30 + 일정 10 = 100점으로 재정의되었다.
North Star
대표 사용자가 3분 안에 현재 검토 가능한 주택형 옵션, 계약금 부족액, 중도금/할부금 부담, 공식 근거, 이번 주 할 일을 이해하고 관심 옵션으로 저장한다.
2. v6.6 대비 v7.0 핵심 변화
영역
v6.6 상태
v7.0 확인
기획 판단
백엔드 라우팅 정리
config/urls.py에 많은 endpoint가 직접 연결되거나 앱 경계가 약하게 보일 수 있었다.
config/urls.py는 include()로 app별 urls.py를 연결한다. profiles, recommendations, notices, notice_docs, funding, ai_coach가 각자의 URL 책임을 가진다. 루트 apps/services.py는 제거되었고 앱별 services만 남았다.
팀이 배운 Django 방식과 일치한다. 기능이 늘어도 URL과 view의 책임을 앱 단위로 찾기 쉬워졌다.
프론트엔드 형식 정리
Vue 3/Vite 구조는 있었지만 TypeScript 흔적과 설정 혼재 리스크가 있었다.
프론트엔드는 Vue + JavaScript + Vite 기준으로 정리되었다. main.js, router/index.js, store/api JS 파일 흐름을 사용하며 .ts, tsconfig, vite.config.ts, vue.config.js, Vue CLI 잔재는 없다. package build도 vite build로 단순화되었다.
팀 학습 범위와 실행 환경에 맞게 정리한 결정이다. 타입 안정성은 줄었지만 협업/이해/수정 속도는 올라간다.
실제 LH/fixture 운영 방향
실제 LH 우선, fixture 보조 fallback 정책이 v6.6에서 정리되었다.
README와 코드에 실제 DB/LH 공고 우선, 소유형 공고가 부족할 때 fixture 보충 정책이 유지된다. 실행/수집/fixture 정책도 최신화되었다.
v7.0에서도 fixture는 기본 데이터가 아니라 안정성 장치다. 화면에서 실제/fixture 출처를 계속 구분해야 한다.
공고 상세 안정화
PDF/공급정보/LLM 기반 옵션의 면적·상세 연결이 일부 불안정할 수 있었다.
LLM/공급정보 기반 옵션에서 0㎡로 보이던 문제를 보정하는 흐름이 추가되었다. 분석 실패 시 상세 화면 전체가 죽지 않도록 처리하고, option_id 기준 자금 계산 연결도 보완되었다.
사용자가 상세 화면에서 막히지 않는 것이 중요하다. 분석 실패는 화면 실패가 아니라 상태 표시와 다음 행동 안내로 처리해야 한다.
현재 검토 가능 공고
실제 LH 공고가 들어와도 과거 마감 공고가 추천 상단에 보일 수 있었다.
마감 지난 공고는 대시보드/추천/캘린더에서 제외한다. /api/notices?active=1이 추가되어 접수 마감일이 오늘 이후인 공고만 조회할 수 있다.
서비스가 “지금 볼 수 있는 후보”를 보여주는 방향으로 정리되었다. 발표와 사용자 경험 모두에서 중요하다.
추천 점수 체계
정책/상품 연계도가 추천 점수에 포함되어 제품 중심축이 흐려질 위험이 있었다.
정책 점수는 제거되고 정책/상품은 보조 정보로 분리되었다. 최종 점수는 자격 35 + 자금 25 + 지역 30 + 일정 10 = 100점이다. 정렬은 총점 우선, option_fit_score는 보조 기준이다.
제품 메시지가 선명해졌다. 추천은 “정책을 많이 붙여서 높은 점수”가 아니라 현재 조건과 공고/옵션 적합도를 보는 것이다.
README/잔재 정리
README 최신화가 필요했고 TypeScript/Vue CLI/정책 점수 잔재가 남을 가능성이 있었다.
README는 실제 LH 수집, fixture fallback, 현재 검토 가능 공고 API, 100점 추천 기준, 마감 공고 제외 기준을 설명한다. apps.services, TypeScript import/type, tsconfig, vue.config.js, vite.config.ts, policy_link 잔재가 없다.
팀원 온보딩과 발표 전 점검에 유리하다. 문서와 코드의 불일치 리스크가 줄었다.
검증 기록
v6.6 기준 test 61개와 build 통과가 기록되었다.
팀 로컬 확인 기준 python manage.py test 63개 OK, npm.cmd run build 통과, 추천 결과 100점 만점 정렬 확인.
v7.0은 구조 변경이 많았으므로 기능 테스트보다 잔재 점검과 회귀 테스트가 특히 중요하다.

3-2. 팀 전체가 반복해야 할 설명 문장
“공고 추천”이 아니라 “공고 안의 주택형 옵션 추천”입니다.
“AI가 판단합니다”가 아니라 “공식 공고문 근거를 AI가 쉽게 설명합니다”입니다.
“당첨 가능성 예측”이 아니라 “준비 가능성 점검”입니다.
“정책·상품까지 다 해주는 서비스”가 아니라 “계약금/중도금/할부금 준비를 돕는 보조 정보를 붙이는 서비스”입니다.
v7.0부터는 “현재 검토 가능한 공고만 추천하고, 정책/상품은 점수가 아니라 보조 정보로 분리한 서비스”라고 설명할 수 있습니다.
v7.0의 코드 구조는 Django app별 URL 분리와 Vue + JavaScript + Vite 기준으로 정리되어 팀원이 더 쉽게 따라올 수 있습니다.

4-1. 핵심 Use Case 흐름
사용자는 프로필에서 자금·청약·희망 면적/가격 조건을 입력한다.
서비스는 실제 LH DB 데이터를 우선 조회하고, DB가 비어 있거나 소유형 공고가 너무 적으면 fixture를 뒤에 보충한다.
마감 지난 공고는 대시보드/추천/캘린더에서 제외하고, 현재 검토 가능한 공고 중심으로 추천한다.
추천 결과에서 서비스 대상 공고와 Top-N 주택형 옵션을 확인한다.
공고 상세에서 공식 공고문 분석 상태, 체크리스트, 주택형 옵션별 근거 문장을 확인한다.
FundingPage에서 선택 옵션의 계약금, 중도금, 입주잔금, 할부금, 융자금, 부족액, 월 준비 목표를 본다.
AI 코치 추천 질문을 눌러 공식 확인 포인트와 이번 주 행동을 묻는다.
관심 공고 또는 관심 주택형 옵션을 저장하고 FavoritesPage에서 다시 비교한다.
4-2. v7.0 완료 기준
사용자는 추천 결과를 보고 “어떤 공고가 좋은지”보다 “어떤 주택형 옵션을 검토할지”를 이해한다.
대시보드/추천/캘린더는 접수 마감일이 지난 공고를 기본 후보에서 제외한다.
화면은 실제 LH API 데이터와 fixture fallback을 구분할 수 있다.
자금 화면에서 중도금 0회/1회/다회차/10회 이상, 할부금, 입주잔금이 NaN·음수·비현실적 0원 오류 없이 표시된다.
추천 점수는 정책 점수 없이 100점 만점으로 계산되고, 정책/상품은 보조 정보로 별도 표시된다.
외부 API와 LLM이 실패해도 fixture/template fallback으로 핵심 흐름이 완주된다.

공고 + HousingUnitOption + 납부안(option_type)

추천 점수
자격 35 + 자금 25 + 지역 30 + 일정 10 = 100점
정책 점수, 상품 점수의 직접 반영
정책/상품은 보조 정보다. 최종 정렬은 total_score 우선, option_fit_score 보조 기준으로 유지한다.
AI 역할
요약, Q&A, chunk 기반 구조화 보조, 공식 근거 설명
당첨 예측, 신청 가능 확정, 대출 승인 판단
AI는 공식 근거와 계산 결과를 설명하는 코치다. 검증 없는 결과 직행은 금지한다.

8. 현재 프로젝트 구조와 코드 분석 요약
항목
v7.0 기준 확인
백엔드 코드 규모
backend/apps 및 backend/config 기준 Python 코드 약 116개 파일, 약 8,800 lines. migrations 제외 기준이며 app별 urls.py가 신설되어 라우팅 경계가 명확해졌다.
프론트 코드 규모
frontend/src 기준 Vue/JS/CSS 코드 약 20개 파일, 약 3,800 lines. TypeScript 파일 없이 JavaScript/Vite 기준으로 정리되어 있다.
테스트 코드
backend/apps/*/tests.py의 test_ 함수 63개 확인. 팀 로컬 기록 기준 python manage.py test 63개 통과.
핵심 백엔드 앱
profiles, notices, notice_docs, recommendations, funding, ai_coach, products, policies, fixture_store.
핵심 프론트 화면
Home, Auth, ProfileForm, Recommendation, NoticeDetail, Funding, Map, AiCoach, Favorites, FloatingCoachChat.
백엔드 라우팅
config/urls.py는 include()로 app별 urls.py를 연결한다. profiles, recommendations, notices, notice_docs, funding, ai_coach가 각각 urls.py를 가진다.
프론트 설정
package.json build는 vite build. vite.config.js는 Vue와 Tailwind Vite plugin을 사용한다. tsconfig, vue.config.js, vite.config.ts 파일은 없다.
잔재 점검
apps.services import 잔재, TypeScript import/type 잔재, policy_link/정책 점수 잔재가 발견되지 않는다. /85 검색 결과는 Tailwind 색상 투명도 표현으로 정상이다.
샘플 PDF
sample_pdfs에 include/exclude PDF 샘플, lh_detail_page.html, expected.json, manifest.json이 존재한다.
검증 기록
팀 로컬 확인 기준 migrate, python manage.py test 63개, check_sample_pdf_regression, npm.cmd run build 통과 및 추천 결과 100점 만점 정렬 확인.

소유형 공공분양 서비스 대상만 DB에 저장한다. 임대형/토지/상가/오피스텔 등은 제외한다.
잔여세대·선착순이라도 공공분양 소유형이면 포함할 수 있다. 애매한 공고는 검토 필요로 분리하는 정책이 필요하다.
--max-pages 기본 5
최신 1~3페이지만 보면 임대형만 들어올 수 있어 기본 탐색 범위를 넓혔다.
소유형 공고가 적은 시기에는 더 늘리거나 readiness report로 범위 밖 정상 제외를 설명한다.
--with-supply-info
LH 공급정보 API로 주택형, 평균분양가격, 대표 면적을 보강하고 임시 HousingUnitOption/PaymentSchedule을 만든다.
정확한 계약금/중도금 회차는 공식 PDF 분석 결과가 우선이다. 임시 옵션은 LH 공급정보 API 기반임을 표시해야 한다.
active=1
접수 마감일이 오늘 이후인 공고만 조회한다.
대시보드/추천/캘린더는 기본적으로 현재 검토 가능한 공고를 사용해야 한다. 전체 목록 확인은 /api/notices로 분리한다.
11-2. Fixture fallback 정책
상황
동작
사용자/발표 의미
DB가 비어 있음
fixture 공고/상품/정책을 사용한다.
처음 실행하거나 API 키가 없어도 핵심 흐름이 비지 않는다.
DB에 실제 공고가 있으나 소유형 서비스 대상이 적음
FIRSTHOME_ENABLE_FIXTURE_SUPPLEMENT=true이고 서비스 대상 수가 FIRSTHOME_MIN_SERVICE_NOTICES보다 작으면 fixture를 뒤에 보충한다.
실제 LH 우선 흐름을 유지하면서 추천 화면이 빈 상태가 되지 않는다.
fixture 보충을 끄고 싶음
FIRSTHOME_ENABLE_FIXTURE_SUPPLEMENT=false로 설정한다.
운영/검증 시 실제 DB 결과만 확인할 수 있다.
마감 지난 공고가 많음
current_notices()와 /api/notices?active=1을 사용한다.
사용자에게 지금 검토 가능한 공고만 보여준다.
운영 원칙: 추천 카드와 상세 화면에서는 data_source를 통해 LH API, fixture, fixture fallback을 구분해야 한다. fixture가 섞이는 것은 실패가 아니라 시연 안정성 장치지만, 사용자에게 공식 분석 결과처럼 보이면 안 된다.
12. 공식 공고문 PDF 분석/회귀 파이프라인
v7.0의 PDF 분석 방향은 v6.6과 같다. 표를 우선 읽고, 납부 구조를 타입으로 구분하며, 틀릴 수 있는 결과는 needs_review로 분리한다. 이번 버전에서는 기능 추가보다 상세 화면 안정성, 면적 보정, option_id 자금 연결을 보완한 것이 중요하다.
12-1. 분석 파이프라인
POST /api/notices/{id}/documents/analyze 요청 또는 analyze_notice_documents 관리 명령으로 분석을 시작한다.
discover_documents_for_notice가 source_url/document_url/file_name 기준으로 문서 후보를 찾는다.
file://, local path, http/https PDF를 resolve하고 원격 PDF는 캐시한다.
pypdf 텍스트 추출과 pdfplumber table row 병합으로 페이지별 텍스트를 만든다.
표 블록이 있으면 공급금액/납부일정 표를 우선 파싱하고, 평문 compact regex는 fallback으로 사용한다.
rules extractor가 주택형/금액/일정/체크리스트를 추출하고, 부족하거나 경고가 있으면 LLM structured extraction을 보조로 시도한다.
validators가 필수 필드, payment schedule, source_text, 금액 합계 등을 검증하고 NoticeExtraction, HousingUnitOption, PaymentSchedule, Checklist, Evidence를 저장한다.
analysis_summary가 verified/needs_review/mock/failed/discovered/excluded 등 사용자 표시 상태로 변환된다.

15. 현재 개발 현황과 남은 리스크
15-1. 구현 완료 또는 1차 완료
백엔드 라우팅을 config/urls.py + app별 urls.py 구조로 정리했다.
루트 apps/services.py를 제거하고 app별 services 책임을 유지했다.

실제 LH 데이터 우선, fixture 보조 fallback 정책을 README와 코드에 유지했다.
공고 상세에서 분석 실패가 전체 화면 실패로 이어지지 않도록 처리했다.
LLM/공급정보 기반 옵션에서 0㎡로 보이는 문제를 보정하는 흐름을 추가했다.
마감 지난 공고를 대시보드/추천/캘린더에서 제외하고 /api/notices?active=1을 추가했다.
추천 점수에서 정책 점수를 제거하고 자격 35 + 자금 25 + 지역 30 + 일정 10 = 100점 체계로 정리했다.
정책/상품은 추천 점수에서 분리해 계약금/중도금 준비 보조 정보로 유지했다.
option_id 기반 FundingPlan, 중도금/할부금 구분 UI, 할부금 요약 badge 흐름을 유지했다.
README에 실제 LH 수집, fixture fallback, 100점 추천 점수, active 공고 API, 마감 공고 처리 기준을 반영했다.
팀 로컬 기록 기준 migrate, python manage.py test 63개, sample PDF regression, frontend build 통과.
